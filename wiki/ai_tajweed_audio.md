# ИИ-проверка таджвида по аудио — исследование (начато 21.07.2026)

## Цель

Долгосрочно снизить нагрузку на устаза при проверке голосовых сдач студентов
(см. [[voice_review]] — сейчас устаз слушает и оценивает таджвид вручную,
ИИ только фиксирует факт проверки). Цель НЕ «заменить устаза разом», а
постепенно отдавать ИИ куски задачи по мере подтверждённой точности.

## Финализированный алгоритм (4 стадии)

Правило: устаз убирается из каждого куска задачи только тогда, когда
точность ИИ на этом куске подтверждена реальными данными — не по календарю,
не разом.

1. **Scribe** — устаз слушает и поправляет как сейчас (голосом). ИИ только
   транскрибирует и структурирует его коррекцию (слово/буква/правило) в БД.
   Побочный эффект — растущий размеченный корпус (студент-аудио → конкретная
   ошибка), бесплатно, без доп. усилий устаза. **Не реализовано.** Требует
   изменения `bot_tg.py:2601-2608` — сейчас голосовые без подписи молча
   отбрасываются (`is_media` и `return`), `file_id` не читается и не
   скачивается вообще. Технически апдейт с voice долетает до бота (privacy
   mode выключен/бот админ в группах) — просто не используется.
2. **Comparative diff** — вместо суждения «в вакууме» по абстрактным
   правилам, модели дают чтение студента + эталонное чтение того же аята,
   просят найти разницу. Заметно точнее Stage-суждения вслепую (см. тесты
   ниже). Требует выровненного эталонного аудио на тот же отрывок.
3. **Confident auto-approve** — как только точность на бакете «явно всё
   верно» подтверждена на реальных данных — ИИ тихо подтверждает такие
   сдачи сам, устаз проверяет только сомнительное/флагованное. Асимметрия
   риска: auto-**подтверждение** — низкий риск (как пропуск редкой ошибки
   самим устазом); auto-**исправление** («ты ошибся, делай так») — высокий
   риск, неверная коррекция активно учит студента неправильно. Поэтому
   исправления всегда идут через устаза, подтверждения — можно тихо.
4. **Расширение автономного покрытия** — по каждому типу ошибки отдельно,
   по мере роста корпуса из Stage 1, с постоянной проверкой согласия с
   устазом, а не разовым переключением.

## Тест 1 (21.07.2026): суждение вслепую — 0/3

Три реальные пары с сегодня: чтение студента + голосовая коррекция устаза
Умар (файлы `audio/Бакыт.m4a` + `Бакыт - Умар.ogg`, аналогично Сатар,
Нурсултан). Модель `google/gemini-2.5-pro` через OpenRouter, аудио → mp3
16kHz mono, `input_audio` content type.

Модели дали чтение студента БЕЗ подсказки, попросили назвать ошибки
таджвида. Отдельно транскрибировали голосовую коррекцию устаза (как
эталон/ground truth). **0 из 3 пар — ни одного совпадения** между тем, что
модель услышала сама, и тем, что реально сказал устаз (по транскрипции).
Ответы модели каждый раз уверенные, детальные, с конкретными
буквами/словами — классический профиль галлюцинации LLM, а не реального
слуха.

**Важная оговорка, поймана вторым проходом ревью**: «ground truth» тут —
ТОЖЕ транскрипция Gemini голоса устаза, не проверена человеком. Сравнение
было модель-vs-модель, а не модель-vs-реальность. **Не верифицировано** —
нужно прослушать хотя бы `Бакыт - Умар.mp3` вручную и подтвердить, точна ли
транскрипция (три пункта: пропущен танвин в «аятин», нет гунны в «ва
май-йатабаддaль», нет сукуна на «аль-китаби»).

## Тест 2 (21.07.2026): сравнение с эталоном — 2/3 (по слову/месту)

Тот же Бакыт, тот же вопрос, но модели дали ДВЕ записи: эталонное чтение
Аль-Хусари (см. ниже) того же отрывка (стр. 17 мусхафа, аяты 106-112) +
чтение студента, попросили найти разницу.

Результат: 2 из 3 пунктов устаза совпали по конкретному слову/месту —
«аятин» (модель: не хватает гунны/идгама на танвине в конце слова — то же
слово, что назвал устаз) и переход «аль-куфра → биль-иман» (та самая фраза
«ва май-йатабаддaль», хоть характер ошибки описан иначе — пропуск vs
отсутствие гунны). «Аль-китаби» не встретился. Заметно лучше, чем 0/3 в
Тесте 1 — сравнительный режим ощутимо эффективнее суждения в вакууме. Та же
оговорка про неверифицированный ground truth применяется и здесь, но два
независимых прогона (разные режимы, разные генерации), сошедшиеся на одних
словах — менее похоже на случайную галлюцинацию.

## Тест 2, продолжение: Сатар и Нурсултан

Тот же режим сравнения с эталоном Хусари, страницы мусхафа подобраны по
словам из Теста 1 (Сатар — стр. 14, аяты 89-93, слово «багьян» из 2:90;
Нурсултан — стр. 12, аяты 77-83, включая явно упомянутый в Тесте 1 аят 83).

- **Сатар**: устаз — пропуск изхара в танвине «багьян» перед гортанной
  буквой. Stage 2 зацепился за ту же фразу («بَغْيًا أَن يُنَزِّلَ»), но
  диагноз другой (сказал про короткую гунну, а по изхару её вообще не
  должно быть). Совпадение по месту, не по диагнозу.
- **Нурсултан**: устаз — 1) «та'ламун» прочитано как «я'ламун» (замена
  буквы), 2) неправильная остановка после «ан-нар». Stage 2 попал **точно**
  в пункт 2 (то же слово, та же формулировка «неверная остановка»). Пункт 1
  — тоже отметил слово «ya'lamūna», но с другим диагнозом (слабый 'айн, не
  замена буквы).

**Паттерн по всем 3 парам**: сравнительный режим стабильно точнее находит
ПРАВИЛЬНОЕ МЕСТО/СЛОВО, чем режим вслепую (Тест 1) — но точный диагноз
(какое именно правило нарушено) совпадает не всегда. Полезный, но
неполный сигнал.

## Источник эталонного аудио — проект Qrum

`D:\dev\qrum\tools\audio_baqara\` — 2748 mp3-файлов, нарезка по страницам
мусхафа и сегментам строк (`page_XXX_first_N.mp3`, `..._first_half.mp3`,
`..._full.mp3` и т.д.), только сура Аль-Бакара пока. Чтец — **Махмуд Халиль
Аль-Хусари** (رحمه الله), `RECITER_ID=6` в `tools/generate_line_audio_v2.py`,
источник — everyayah.com / quran.com API (`mirrors.quranicaudio.com/everyayah/
Husary_64kbps/`).

Соответствие аят → страница — в `D:\dev\qrum\assets\quran\pages.json`
(список объектов `{page, surah, ayah, firstHalf, secondHalf, ...}`, `surah` —
русское название типа «Аль-Бакара», `ayah` — диапазон строкой типа
«106–112»). Разбивка на сегменты НЕ строго по-аятная (по частям строки
мусхафа), может потребовать доп. работы для точного маппинга аят→сегмент
при масштабировании за пределы «плюс-минус страница».
