# Reasoning-модели и max_tokens — обрыв текста

> Зафиксировано: 01.07.2026. Найдено при переключении `_NASIHA_MODEL` на `deepseek/deepseek-v4-pro`.

## Суть проблемы

`deepseek-v4-pro` (и, вероятно, другие "pro"/reasoning-модели на OpenRouter) тратят токены
из `max_tokens` на **скрытое рассуждение** ещё до того, как начнут писать видимый ответ.
Если рассуждение выходит длинным, лимит съедается целиком — ответ обрывается на полуслове
(`finish_reason=length`), а в худшем случае возвращается **полностью пустая строка**
(рассуждение съело весь бюджет, на текст не осталось ничего).

`deepseek-v4-flash` и `z-ai/glm-5.2` (модели, что стояли раньше) этой проблемы не показывали.

## Как проверять при смене модели на "pro"/reasoning-тир

Прямой вызов OpenRouter API (не через `_or_call`, а сырым запросом) — смотреть
`finish_reason` и `usage.completion_tokens`:

```python
result = call(messages, max_tokens, model)
result["choices"][0]["finish_reason"]      # "stop" — ок, "length" — обрезано
result["usage"]["completion_tokens"]        # если == max_tokens — почти наверняка обрыв
```

Прогнать 5+ раз — проблема проявляется не всегда, зависит от того, насколько модель
«разгонится» на рассуждении в конкретном запросе.

## Практическое правило

Для ТДБР-контента (насыха, тадаббур-пост, короткие AI-фразы) на `deepseek-v4-pro`:
`max_tokens` не ниже **1600**, даже для одной короткой фразы (эпиграф на 15-20 слов
всё равно может упереться в лимит на рассуждении при `max_tokens=150`).

Проверено эмпирически 01.07.2026: 20+ живых вызовов подряд на `max_tokens=1600` — 0 обрывов.

## Где применено

`core/ai.py`: `daily_tadabbur_post`, `daily_nasiha`, `morning_report_intro`,
`group_motivation_base` (параметр `max_tokens`, для ТДБР передаётся 1600).

См. также [[project_decisions]] — модель `_NASIHA_MODEL` переключалась на pro 01.07.2026
именно из-за качества ТДБР-контента.
