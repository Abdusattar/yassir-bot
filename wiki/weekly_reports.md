# Еженедельные отчёты: устазам и студентам

## Принцип: разные аудитории — разные метрики

Два отдельных еженедельных отчёта, каждый в свой чат, с чётко разделённым
содержанием. Разделение появилось не сразу — первая версия смешивала
операционные метрики (сколько групп провело урок) со студенческой сводкой,
устаз поправил: «Тадаббур это студенты», операционка — только устазам.

## `weekly_ops_report` — устазам, «Масштабирование»

- **Когда**: понедельник 07:00, с подстраховкой во вторник (если бот не успел
  в понедельник — упал, деплой и т.п.). Флаг `bot_settings["ops_report_week"]`
  (ключ — дата понедельника отчётной недели) не даёт задвоить отправку.
- **Поглотил** более раннюю версию `weekly_lesson_report` (список групп без
  урока за неделю) — устаз сам попросил объединить, увидев первую версию.
- **Метрики недели** (все со сравнением к прошлой неделе, ↑/↓):
  - 🕌 Уроки: X/N групп отметились «у»/«u» за неделю (`category='attendance'`
    в `score_events`) + список тех, кто не провёл
  - 📝 Сдачи заданий: сырое количество `score_events` с `category='task'`
  - 🎙 Проверка таджвида: % проверенных `voice_submissions`, плюс **3 худшие
    группы по факту** (без искусственного порога)
- **Порог «просела <80%» — отброшен**. Первая версия флагала группы с % ниже
  80 — на реальных данных сервера под порог попали **17 из 17** групп (общий
  % проверки — 56%), список был бесполезен («просела проверка: все»). Заменено
  на простой топ-3 худших по реальной цифре — без ложного ощущения нормы.
- N (знаменатель уроков) включает ВСЕ нетадаббур-группы, в т.ч. административные
  («Отбор.», «Группа для детей», «Yassir подготовительная») — устаз подтвердил,
  фильтровать не нужно.

## `weekly_tadabbur_summary` — студентам, «Тадаббур»

- **Когда**: вторник 12:00. Изначально предлагался понедельник (вместе с
  ops_report) — устаз указал, что в Тадаббур и так 3 ежедневных сообщения по
  утрам (`morning_tadabbur_report` 07:00, `tadabbur_nasiha` 09:00,
  `tadabbur_invite_reminder` 10:00) — сводка потерялась бы в кластере. Развели
  на другой день/время намеренно.
- **Содержание** (никакой операционки, только то, что важно самим студентам):
  - Совокупная сдача за неделю (сколько всего заданий сдано всеми, командная
    цифра, не сравнение групп)
  - 🌟 «Примеры недели» — сдавали **ВСЕ** задания **ВСЕ 7 дней** недели подряд.
    Заголовок изначально был «Герои недели» — устаз попросил менее пафосный.
    На практике критерий очень строгий: первый прогон на реальных данных дал
    **1 человека** из ~100+ студентов — это ожидаемо, не баг.
  - 🔥 «Держат планку» — стрик 7+ дней (`get_streak_days`, якорь — последний
    день отчётной недели, не «сегодня»)
  - Закрытие — аят или хадис (см. ниже)
- **Исключение из списков — только личный ID устаза**: `SUPER_ADMIN_IDS[0]`
  (Абдусаттар), а НЕ все устазы/супер-админы. Первая версия фильтровала всех
  админов — устаз явно поправил: другие устазы имеют право попасть в список,
  если реально заслужили, исключать нужно только его самого.

## Закрывающий аят/хадис — «без AI», но с оговоркой

Устаз сначала попросил закрытие «без AI» — просто кешированный перевод.
На сухом прогоне против прод-копии выяснилось: `sample_hadith()`/`sample_ayah()`
(`core/sampler.py`) **сначала выбирают из непереведённых** записей (чтобы не
повторяться), и на момент теста пул непереведённых (66 хадисов + 245 аятов)
**не пересекался вообще** с уже закешированными русскими переводами — чистое
«без AI» давало пустую секцию **0 раз из 20** попыток.

Решение: пробуем оба типа (хадис/аят), для найденного — сначала кеш
(`sampler.get_cached_translation`/`get_cached_ayah_translation`), если пусто —
**один разовый AI-перевод** (`core.ai._get_hadith_translation`/
`_get_ayah_translation`, уже существующие приватные функции модуля `ai.py`)
с сохранением в кеш. Раз в неделю, не ежедневная AI-нагрузка. Если оба типа
не дали текста — секция просто пропускается (не шлём «сырой» арабский/
английский текст студентам).

## Практика: сухой прогон перед деплоем контентных фич

Обе фичи проверялись не только синтаксически (`pytest`), а **против копии
реальной прод-БД на сервере** — временная папка `/tmp/yassir_dryrun` (копия
`core/` + `config.py` + `sources/` + копия `.db`, не сама БД), вызывались
только чистые/read-only функции напрямую, без `send_message`/`set_setting`.
Поймал реальные проблемы с контентом (пустые секции, бесполезный порог), не
видные по коду на глаз. Папка удалена после теста, ничего не ушло в реальные
чаты до подтверждения устаза финальным превью.
