# Подготовительная группа (prep)

## Суть

Испытательный период: студент вступает в подготовительную группу.
Если за 14 дней набрал ≥5 дней отчётов — **переходит сразу, как только набрал**
(не дожидаясь 14-го дня) в русскоязычную relaxed-группу — ссылка в личку.
Если <5 дней за 14 — переводится в Тадаббур (гарантированно, с проверкой) и
физически убирается из prep-чата.

Задания те же что в relaxed (m, r, t) — **все три обязательны**, включая
«Повторение», хотя студенты только начали. Решение устаза 19.07: дисциплину
лучше прививать сразу, а не делать для prep исключение.

## Параметры (core/prep.py)

```python
PREP_DAYS = 14       # дней на испытание (крайний срок для ветки провала)
PREP_MIN_DAYS = 5    # минимум дней с отчётом для перехода
```

## Расписание (scheduler.py)

| Время | Действие |
|---|---|
| 07:00 | `send_prep_reminders()` — напоминание в prep-группу (активные/пассивные) |
| 21:00 | `check_prep_students()` — ежедневная проверка по всем активным |

## Флоу 21:00 — check_prep_students (переписан 19.07.2026)

`get_prep_students_active()` возвращает **всех** активных студентов prep
(без фильтра по дням — раньше фильтровало строго `elapsed>=14`, из-за чего
студент, набравший 5 дней уже на 6-й день, всё равно ждал предложения до
14-го). Для каждого:

1. `count_report_days_since(uid, group_id, joined_date)` — сколько дней с
   отчётом (задание любого типа — считается сам факт хотя бы одной сдачи в
   этот день, не все три сразу)
2. **≥5 дней** (в любой момент, хоть на 6-й день, хоть на 13-й):
   - Если уже предлагали переход этому студенту (маркер, см. ниже) — `continue`,
     ждём, пока подтверждённо вступит по уже отправленной ссылке
   - Иначе: находит ru-relaxed-группу через `get_relaxed_groups_by_lang("ru")`
     (наименьшую по числу студентов, у которой сохранена `invite_link`),
     ставит маркер, шлёт поздравление + ссылку **прямо в личку, без выбора
     языка** (см. «Выбор языка убран» ниже), записывает ожидание в
     `prep_graduates`, добавляет в Тадаббур как страховку
3. **<5 дней, elapsed≥14** (крайний срок истёк): деактивирует из prep,
   гарантирует Тадаббур, физически кикает из чата (см. «Провал» ниже)
4. **<5 дней, elapsed<14** (ещё есть время): ничего не делает, ждёт

## Дедуп предложения — score_events category='prep_offer'

Раньше состояние «уже предложили» жило только в памяти
(`_pending_lang_choice`, терялось при рестарте бота). Теперь — персистентный
маркер в БД: при первом предложении `add_bonus(uid, group_id, joined_date, 0,
"prep_offer")` (0 баллов, просто факт). Проверка перед повторным
предложением — `_has_prep_offer()`, `SELECT 1 FROM score_events WHERE
category='prep_offer'`. Переживает рестарт бота, не спамит повторно, пока
студент решает, переходить или нет.

## Выбор языка убран (19.07.2026, решение пользователя)

Раньше при переходе бот показывал 2 inline-кнопки (кыргызский/русский),
`handle_prep_callback` обрабатывал нажатие и подбирал relaxed-группу нужного
языка. Причина отмены: G-8 — единственная кыргызоязычная relaxed-группа в
пуле (с сохранённой `invite_link`), и в базе у неё нет `role='admin'`
(устаз не назначен через бота). Убирать G-8 из пула означало оставить
кыргызоязычных выпускников совсем без группы (`get_relaxed_groups_by_lang`
вернул бы пустой список). Решение: выбор убрать полностью, все выпускники
prep всегда идут в русскоязычную группу.

Убрано целиком: `handle_prep_callback`, `_pending_lang_choice` (in-memory
состояние), `callback_query`-роутинг в `bot.py` (включая `callback_query` из
`allowed_updates` — в боте больше нет inline-кнопок нигде), i18n-ключи
`prep_link_ru`/`prep_link_ky`/`prep_no_group`.

**G-8 остаётся открытым вопросом** — не проверяли, есть ли у неё устаз в
реальности, просто не назначен через `/admin`.

## Подтверждённое вступление — announce_prep_graduate_arrival

Без изменений с 13.07. Вызывается из `bot.py` в обоих местах обработки входа
участника в чат (по инвайт-ссылке `chat_member` и через `new_chat_members`),
после успешного `add_student(...)` для уже известного пользователя:

1. `pop_prep_graduate(phone, group_id)` — если для ЭТОЙ группы есть ожидание,
   забирает запись (и удаляет её)
2. Если нашлась — только теперь `_deactivate_from_prep_by_group_id()`
   (деактивация из старой prep-группы)
3. Объявление в новом чате: `prep_graduate_announce_new` (обращение к устазу и группе)
4. Объявление в старом (prep) чате: `prep_graduate_announce_old` (дуа за уходящего)
5. Если студент так и не перешёл — остаётся активным в prep, при следующей
   ежедневной проверке `_has_prep_offer` вернёт True → просто ждём дальше
   (не переспрашиваем повторно)

Если запись не найдена (обычный новый участник, не выпускник prep) —
функция тихо ничего не делает.

## Провал (<5 дней за 14) — переписан 19.07.2026

Раньше: только `active=0` в БД, физически студент оставался в чате. Реальный
инцидент — Ринат: провалил 09.07, написал отчёт в prep-чате 11.07 → дыра
авторегистрации (`handlers.py`) тихо вернула его активным студентом.

Теперь, в порядке:
1. `_deactivate_from_prep()` — `active=0`
2. Проверка членства в Тадаббуре (`find_by_phone`) — если не состоит,
   `add_student()` добавляет (страховка, раньше была только для ветки
   успеха — закрыт «известный пробел», см. ниже)
3. Личное сообщение `prep_failed` — если студента только что добавили в
   Тадаббур, к тексту добавляется реальная ссылка-приглашение (`👉 {link}`),
   иначе сообщение «остаёшься в Тадаббуре» было бы просто текстом без
   возможности его выполнить
4. Групповое сообщение `prep_failed_group` в сам prep-чат — имя + сколько
   дней сдано из скольких. Раньше об этом не знал никто, кроме самого
   провалившего — теперь видно всем, для ощущения, что правила
   применяются одинаково
5. Физический мягкий кик (`ban_member`+`unban_member`) — закрывает дыру
   авторегистрации именно для этого случая (не для дыры в целом, см. ниже)

## Смежная проблема — авторегистрация (общая дыра, НЕ prep-специфична)

Два места вызывают один и тот же паттерн: `bot.py` (chat_member-хендлер —
вход по инвайт-ссылке) и `core/handlers.py` (первое сообщение в чате, когда
телефон уже известен системе). Если студент, деактивированный из ЛЮБОЙ
группы (провал prep, перевод в Тадаббур за неактивность, ручное удаление),
физически остаётся в чате и потом пишет туда сообщение — бот находит его
глобальную запись в `users` и молча возвращает студентом через
`add_student()`, минуя устаза.

**19.07.2026: prep-специфичный инстанс закрыт** — теперь при провале prep
студента физически кикает (см. выше), так что для ЭТОЙ ветки дыра больше не
актуальна (человека просто нет в чате, чтобы написать туда снова).

**Общая дыра вне prep всё ещё открыта** — переводы в Тадаббур за
неактивность (`core/transfers.py`) кикают с 13.07, но если человек вернётся
в чат другим путём (повторно вступит по инвайт-ссылке) и напишет — дыра
сработает снова. Решение пользователя 15.07.2026 «пускай дыра остаётся,
пока» не пересматривалось. Идея команды `/promote` для ручного возврата
студента в обход порога дней — формат ещё не выбран.

## Напоминание 07:00 — send_prep_reminders

Без изменений. Для каждой prep-группы одно сообщение:
- **Активные** (сдавали ≥1 день): список имён
- **Пассивные** (0 дней): список имён

Тексты: `prep_reminder_active`, `prep_reminder_inactive` из i18n.

## Таблица prep_graduates (персистентно в БД)

```sql
prep_graduates(phone, target_group_id, name, from_group_id, from_chat_id, created_date)
PRIMARY KEY(phone, target_group_id)
```

Хранится между отправкой ссылки и подтверждённым вступлением в целевую
группу (может быть дни/недели). `add_prep_graduate()` / `pop_prep_graduate()`
в `core/db.py`.

## invite_link

- Хранится в `groups.invite_link` (TEXT)
- Устаз сохраняет: `/setlink https://t.me/+xxx`
- `get_relaxed_groups_by_lang(lang)` возвращает relaxed-группы с
  `invite_link`, sorted по количеству студентов ASC — с 19.07 вызывается
  только с `"ru"` (выбор языка убран)

## Состав prep-чата — разбор 19.07.2026 (пример, не общая закономерность)

На живом кейсе (группа id=14) при проверке «сколько студентов на самом деле»
всплыло: Bot API даёт только `getChatMemberCount`/`getChatAdministrators`
(полного списка участников для группы Telegram не отдаёт). В чате из 11
человек только 3 были реальными активными студентами — остальные:
3 давно неактивных бывших студента (провалили prep ещё в июне, физически
не выгнаны — вычищены вручную), 1 числящийся `role='admin'` в БД без
реального Telegram-admin статуса (визуально неотличим от студента в
интерфейсе), 1 реальный устаз другой группы, физически в чате, но не
привязанный в БД к этой группе, и 1 участник, не найденный ни в `users`,
ни в `unregistered_members` — бот не видел его вступление вообще (Bot API
не даёт способа его идентифицировать по имени).

Урок: деактивация в БД (`active=0`) не равна физическому отсутствию в
Telegram-чате — эти два состояния надо сверять отдельно, автоматика (кроме
только что зачиненных путей выше) массово это не делает.
