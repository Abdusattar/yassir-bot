# Подготовительная группа (prep)

## Суть

Испытательный период: студент вступает в подготовительную группу, у него есть 14 дней.  
Если сдаёт ≥5 дней отчётов — выбирает, в какую relaxed-группу перейти (по языку), и
**остаётся активным в prep до подтверждённого вступления** в целевую группу (13.07 фикс,
см. ниже).  
Если <5 дней — деактивируется из prep сразу; предполагается, что остаётся в Тадаббуре, но
это не гарантировано кодом (см. «Известный пробел»).

Задания те же что в relaxed (m, r, t).

## Параметры (core/prep.py)

```python
PREP_DAYS = 14       # дней на испытание
PREP_MIN_DAYS = 5    # минимум дней с отчётом для прохода
```

## Расписание (scheduler.py)

| Время | Действие |
|---|---|
| 07:00 | `send_prep_reminders()` — напоминание в prep-группу |
| 21:00 | `check_prep_students()` — проверка у кого истёк срок |

## Флоу 21:00 — check_prep_students

1. `get_prep_students_due()` — студенты в prep-группах, joined_date ≥ 14 дней назад
2. `count_report_days_since(uid, group_id, joined_date)` — сколько дней с отчётом
3. **≥5 дней**: `_send_choice()` — поздравление + inline кнопки выбора языка
4. **<5 дней**: `_deactivate_from_prep()` — деактивация + сообщение `prep_failed`
   (немедленно, эта ветка фиксом 13.07 не затронута)

## Выбор языка (callback_query)

- Кнопки: "Кыргызский 🇰🇬" → `prep_lang:ky`, "Русский 🇷🇺" → `prep_lang:ru`
- Обработчик `handle_prep_callback(cq)` в `core/prep.py`
- Находит наименьшую relaxed-группу с нужным языком у которой есть invite_link
- **НЕ деактивирует студента из prep сразу** (фикс 13.07 — раньше деактивировал тут же,
  и если студент игнорировал ссылку, оставался вообще без активной группы — реальный
  случай на проде)
- Отправляет ссылку на группу в личку студенту
- Сразу добавляет студента в Тадаббур как страховку (`add_student`) — на случай если
  ссылкой так и не воспользуется
- Записывает ожидание в `prep_graduates` (`add_prep_graduate`) — phone, целевая группа,
  имя, откуда (group_id + chat_id)

## Подтверждённое вступление — announce_prep_graduate_arrival (13.07)

Вызывается из `bot.py` в обоих местах обработки входа участника в чат (по инвайт-ссылке
`chat_member` и через `new_chat_members`), после успешного `add_student(...)` для уже
известного пользователя:

1. `pop_prep_graduate(phone, group_id)` — если для ЭТОЙ группы есть ожидание, забирает
   запись (и удаляет её)
2. Если нашлась — только теперь `_deactivate_from_prep_by_group_id()` (деактивация из
   старой prep-группы)
3. Объявление в новом чате: `prep_graduate_announce_new` (обращение к устазу и группе)
4. Объявление в старом (prep) чате: `prep_graduate_announce_old` (дуа за уходящего)
5. Если студент так и не перешёл — просто остаётся активным в prep и на следующих
   ежедневных проверках снова получит `_send_choice()` (не теряется)

Если запись не найдена (обычный новый участник, не выпускник prep) — функция тихо
ничего не делает.

## Напоминание 07:00 — send_prep_reminders

Для каждой prep-группы одно сообщение:
- **Активные** (сдавали ≥1 день): список имён
- **Пассивные** (0 дней): список имён

Тексты: `prep_reminder_active`, `prep_reminder_inactive` из i18n.

## in-memory состояние

```python
_pending_lang_choice: dict = {}  # {uid: {group_id, chat_id, name, glang, joined_date}}
```

Заполняется в `_send_choice`, читается в `handle_prep_callback`.  
Живёт в памяти (не в БД) — при перезапуске бота теряется (студент просто получит
предложение заново на следующей ежедневной проверке, не критично).

## Таблица prep_graduates (13.07, персистентно в БД)

```sql
prep_graduates(phone, target_group_id, name, from_group_id, from_chat_id, created_date)
PRIMARY KEY(phone, target_group_id)
```

В отличие от `_pending_lang_choice`, живёт в БД — переживает рестарт бота. Хранится
между выбором языка и подтверждённым вступлением в целевую группу (может быть дни/недели).
`add_prep_graduate()` / `pop_prep_graduate()` в `core/db.py`.

## invite_link

- Хранится в `groups.invite_link` (TEXT)
- Добавлен через `ALTER TABLE groups ADD COLUMN invite_link TEXT` при `init()`
- Устаз сохраняет: `/setlink https://t.me/+xxx`
- `get_relaxed_groups_by_lang(lang)` возвращает relaxed-группы с invite_link, sorted по количеству студентов ASC

## Известный пробел (не закрыт 13.07)

Ветка «<5 дней» (`prep_failed`) предполагает, что студент уже состоит в Тадаббуре
(добавляется туда при первичной регистрации, если физически вступил по ссылке в чат
Тадаббура) — но это не гарантировано кодом. Реальный случай на проде (Махмуд, 26.06):
никогда не вступал в чат Тадаббура → после провала prep остался вообще без единой
активной группы. Страховку (`add_student` в Тадаббур) добавили 13.07 только для ветки
«прошёл» (`handle_prep_callback`), не для «не прошёл» (`_deactivate_from_prep` в
`check_prep_students`). Кандидат на отдельный фикс.

## Смежная проблема — авторегистрация (сознательно не решена)

Два места вызывают один и тот же паттерн: `bot.py` (chat_member-хендлер — вход по
инвайт-ссылке, лог `"chat_member: added existing user %s to group %s"`) и
`core/handlers.py:641-646` (первое сообщение в чате, когда телефон уже известен
системе). Если студент, деактивированный из ЛЮБОЙ группы (провал prep, перевод в
Тадаббур за неактивность в `core/transfers.py`, ручное удаление), потом заходит в чат
той же группы или пишет туда сообщение — бот находит его глобальную запись в `users`
(активна всегда) и молча возвращает его студентом в ту же группу через `add_student()`
— минуя устаза, минуя весь предусмотренный сценарий перевода. Устаз не участвует в
решении «можно ли этому человеку вернуться».

Реальные случаи: Ринат (uid 105) — провалил prep 09.07, написал отчёт в подготовительной
11.07 → тихо вернулся туда же. Камаль (uid 81, N-1) — переводился в Тадаббур за
неактивность и кикался ботом **4 дня подряд** (12–15.07.2026), каждый раз возвращаясь
в N-1 через эту же дыру.

**15.07.2026**: опасное следствие дыры устранено — раньше молчаливый возврат оставлял
студенту старый (плохой) счётчик пропусков, и на следующей ежедневной проверке его
кикало снова, он возвращался через дыру, и так по кругу бесконечно (кейс Камаля).
Теперь `add_student()` сбрасывает `user_groups.joined_date` при любой (пере)активации
— см. [[skip_counting]] — так что молчаливый возврат больше не запускает бесконечный
цикл кик/возврат, человек просто корректно становится студентом заново.

Сама дыра (бот решает за устаза, пускать ли человека обратно) **осознанно оставлена
как есть** — решение пользователя 15.07.2026: «пускай дыра остаётся, пока». Отдельная
сессия по этой теме уже обсуждалась 13.07 (там же — идея команды `/promote` для
ручного возврата студента в обход порога дней, формат не выбран).
