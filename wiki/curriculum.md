# Программа обучения по частям (нахв/таджвид)

## Принцип

Устаз хочет, чтобы бот вёл студентов по программе нахва и таджвида —
маленькими частями, по расписанию, а не одним большим текстом сразу.
Решение от 08.07.2026.

## Почему не генерация на лету

ИИ не имеет доступа к реальному тексту книг во время ответа студенту —
только к своей обучающей памяти. Генерировать урок по религиозной теме
«на лету» и сразу слать студентам — тот же риск неточности, что мы весь
день чинили в AI-проверке (харакаты в муфрадате, лишние замечания в
таджвиде), только проактивный и незаметный. Поэтому — два этапа.

## Источники

  49 глав, 261 тема. Визуально в таблице 3 колонки (глава+счётчик уроков
  в главе → тема → число); при чтении через python-docx колонка «тема»
  дублируется из-за horizontal-merge (gridSpan=2), из-за чего казалось,
  что колонок 4. Третья визуальная колонка (число) = число частей на
  тему — подтверждено устазом 08.07.2026. Плюс книги-первоисточники: «Аль-Аджуррумия»
  (Ибн Аджуррум) и «Ан-Нахв аль-Вадых» (Али аль-Джарим, Мустафа Амин) —
  обе свободно доступны (Internet Archive и др.), реальный текст, не
  придуманный.
- **Таджвид**: разбивку на темы/главы делает Claude (не устаз, зелёный
  свет от устаза 08.07.2026) — по «Аль-Мукаддима аль-Джазарийя» (Ибн
  аль-Джазари) и «Иллюстрированный таджвид» (Айман Рушди Сувейд).
  Начата глава «مخارج الحروف» (места выхода букв) — первые 3 темы

  устазу на одобрение 08.07.2026. Остальные главы (сифат, ахкам нун-
  сакина/мим-сакина, мадд, гунна, калькаля, лям/ра) — не начаты.

## Таблица `curriculum_parts`

`subject` (`n`/`j`), `chapter`, `topic`, `part_number`/`part_total`,
`order_index` (очередь), `content`, `review_chat_id`/`review_message_id`,
`approved_at`, `published_at`.

## Одобрение устазом — отключено (08.07.2026)

Изначально: часть готовилась заранее, шла устазу Умару в личку
(`config.CURRICULUM_REVIEWER_ID`) на одобрение (👍 или любой ответ —
`handle_reaction`/`process_message` в `core/handlers.py`), публикация
ждала `approved_at`. Устаз явно попросил убрать блокировку: «это не
Коран/хадисы/хукмы, а грамматика и таджвид, будут ошибки — устаз мне
скажет» + «он и так в группах — незачем слать отдельно». Публикация
теперь идёт по расписанию без ожидания одобрения. Механизм одобрения
(`get_pending_curriculum_review_by_chat`, `mark_curriculum_approved*`,
ветка в `process_message`/`handle_reaction`) оставлен в коде рабочим,
но не задействован — на случай если решение снова изменится.

## Расписание (`core/scheduler.py`)

- `request_curriculum_review()` — раньше вызывалась ежедневно в 07:00
  (буфер на 3 части вперёд в личку устазу), сейчас **не вызывается**
  из планировщика (устаз видит опубликованное прямо в группах).
  Функция осталась в файле нетронутой.
- `publish_curriculum_parts()` — раз в неделю, **четверг 08:00**. Первый
  запуск — 09.07.2026 («день 1»), дальше «через неделю» — подтверждено
  устазом 08.07.2026. Тема из N частей естественно растягивается на N
  недель («есть темы одна неделя, есть две-три», его слова). Берёт
  следующую **не опубликованную** часть по `order_index` (без проверки
  `approved_at`) и шлёт во все группы, где в `tasks` есть соответствующая
  буква (`n`/`j`), кроме тадаббур-групп. Нет частей в очереди —
  безопасный no-op, ничего не публикует.

## Подсчёт объёма (22.07.2026)

Пересчитан весь docx-план по нахву: **49 глав** (совпало с уже зафиксированным здесь числом —
хорошая перекрёстная проверка подсчёта), **≈420 частей/уроков** суммарно по всем главам (число
рядом с темой в оглавлении — не страницы, а число частей на тему). При темпе 1 часть/неделю —
это больше 8 лет на один нахв, без учёта таджвида (для таджвида такой детальной раскладки по
главам нет — известны только начатые/неначатые главы, см. выше). Точность подсчёта ±10-15%
(таблица в docx криво разворачивается в плоский текст через python-docx/XML, поэтому границы
некоторых секций пришлось восстанавливать по сумме дочерних пунктов).

## Очередь на 22.07.2026 — что реально написано

- **Нахв**: order_index 1 (`أنواع الكلمة`, опубликовано 09.07) + **4 новые части** написаны
  22.07 (order_index 2-5): `اسم وعلامته` (2 части, как и положено по плану — не 1, как было в
  первой неудачной попытке), `الاسم المفرد`, `الاسم المثنى`. Со сверенными цитатами матна

  четырёх — уйдут по одной части в четверг начиная с 23.07. Хватит на 4 недели.
- **Таджвид**: order_index 3 (`الشفتان`/«Губы») в очереди, последняя, уйдёт 23.07. После
  этого обе очереди снова пусты — темы بناء/إعراب (следующая глава нахва, ~37 частей) и
  сифат/ахкам-нун-сакина и т.д. (таджвид) не начаты.

### Урок про источники (22.07.2026)

Точечные `WebSearch`-запросы по конкретной теме («علامات الاسم», «إعراب المثنى») надёжно
возвращают дословные цитаты матна «Аль-Аджуррумия» — сработало для всех 4 частей выше.
А вот `WebFetch` с просьбой скопировать **несколько глав целиком за один запрос** — отказывает
(fair use), даже для текста, который сам по себе давно в общественном достоянии. Для следующей
главы (بناء/إعراب, ~37 частей) браться писать сразу всё нельзя — на пробных запросах по этой
теме поиск начал отдавать пересказы с вторичных сайтов (mawdoo3, alukah, wikiversity) вместо
самого матна. Делать по темам точечно, как в первый раз, не пытаться охватить сразу всю главу
одним broad-запросом.

### Первая попытка была неудачной — для истории

22.07 сначала написал 3 части не проверив эту страницу вики заранее — использовал короткий
формат (~900-1200 симв.), который устаз уже раньше явно раскритиковал как недостаточный (см.
раздел «Осознанно не сделано» ниже), не разбил «اسم وعلامته» на 2 части как в плане, и написал
по памяти вместо реальных цитат. Откатил (id 5-7, `DELETE WHERE published_at IS NULL` — ничего
не успело уйти студентам) и переписал правильно (id 8-11, описаны выше). Из-за этого в памяти
Claude Code (`feedback_check_wiki_before_acting.md`) теперь зафиксировано правило: проверять
профильную страницу вики ДО начала работы, не только при отладке.

## Осознанно не сделано

- Прогресс общий по всем группам с нужным заданием, не по каждой группе
  отдельно (устаз подтвердил — 08.07.2026).
- Нет отзыва/правки уже опубликованной части — устаз про первую часть
  сказал «давайте одобрим, потом посмотрим», это принято к сведению,
  без доработки (08.07.2026).
- Контент по всем темам должен быть содержательнее, ближе к объёму
  реальной книги — устаз отметил, что первые черновики были слишком
  короткими (08.07.2026). При написании новых частей ориентироваться
  на расширенный формат (см. пример в истории сессии), не на исходный
  сжатый пилот.
