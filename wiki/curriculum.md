# Программа обучения по частям (нахв/таджвид)

## Принцип

Устаз хочет, чтобы бот вёл студентов по программе нахва и таджвида —
маленькими частями, по расписанию, а не одним большим текстом сразу.
Решение от 08.07.2026.

## Почему не генерация на лету

ИИ не имеет доступа к реальному тексту книг во время ответа студенту —
только к своей обучающей памяти. Генерировать урок по религиозной теме
«на лету» и сразу слать студентам — тот же риск неточности, что мы весь
день чинили в AI-проверке (харакаты в муфрадате, лишние замечания в
таджвиде), только проактивный и незаметный. Поэтому — два этапа.

## Источники

- **Нахв**: `knowledge/нахв/الكلمة جاهز.docx` — готовый план устаза,
  49 глав, 261 тема. Визуально в таблице 3 колонки (глава+счётчик уроков
  в главе → тема → число); при чтении через python-docx колонка «тема»
  дублируется из-за horizontal-merge (gridSpan=2), из-за чего казалось,
  что колонок 4. Третья визуальная колонка (число) = число частей на
  тему — подтверждено устазом 08.07.2026. Плюс книги-первоисточники: «Аль-Аджуррумия»
  (Ибн Аджуррум) и «Ан-Нахв аль-Вадых» (Али аль-Джарим, Мустафа Амин) —
  обе свободно доступны (Internet Archive и др.), реальный текст, не
  придуманный.
- **Таджвид**: разбивку на темы/главы делает Claude (не устаз) — по
  «Аль-Мукаддима аль-Джазарийя» (Ибн аль-Джазари) и «Иллюстрированный
  таджвид» (Айман Рушди Сувейд). Пока не сделано — блокирует контент
  по таджвиду, инфраструктура уже готова.

## Таблица `curriculum_parts`

`subject` (`n`/`j`), `chapter`, `topic`, `part_number`/`part_total`,
`order_index` (очередь), `content`, `review_chat_id`/`review_message_id`,
`approved_at`, `published_at`.

## Одобрение устазом

Часть готовится заранее (не в реальном времени), шлётся устазу Умару
в личку (`config.CURRICULUM_REVIEWER_ID`, по умолчанию второй
`SUPER_ADMIN_ID`). Устаз ставит 👍 на сообщение — реакция ловится тем
же механизмом, что и проверка голосовых (`handle_reaction`,
`core/handlers.py`), только для личных чатов вместо групп: если
`get_group(chat_id)` не находит группу — значит это личка, ищем
совпадение по `review_chat_id`+`review_message_id` в `curriculum_parts`
и помечаем `approved_at`. Doп. проверка на админство не нужна — само
совпадение id уже означает, что это тот самый черновик, который бот
отправил.

## Расписание (`core/scheduler.py`)

- `request_curriculum_review()` — ежедневно в 07:00. Если у устаза нет
  частей на рассмотрении меньше буфера (`_REVIEW_BUFFER = 3`,
  `count_pending_curriculum_review`) — досылает недостающие. Копия
  черновика уходит и остальным `SUPER_ADMIN_IDS` (просто для информации —
  на одобрение влияет только 👍/любой ответ от `CURRICULUM_REVIEWER_ID`
  в его личке).
- `publish_curriculum_parts()` — раз в неделю, **четверг 08:00**. Первый
  запуск — 09.07.2026 («день 1»), дальше «через неделю» — подтверждено
  устазом 08.07.2026. Тема из N частей естественно растягивается на N
  недель («есть темы одна неделя, есть две-три», его слова) — отдельного
  правила «2 темы в месяц» нет, это просто следствие разной длины тем.
  Берёт следующую **одобренную, но не опубликованную** часть и шлёт во
  все группы, где в `tasks` есть соответствующая буква (`n`/`j`), кроме
  тадаббур-групп. Нет одобренной части — безопасный no-op, ничего не
  публикует.

## Осознанно не сделано

- Прогресс общий по всем группам с нужным заданием, не по каждой группе
  отдельно (устаз подтвердил — 08.07.2026).
- Что делать, если к понедельнику часть не одобрена — пока просто
  пропускаем неделю (публикация no-op). Отдельного напоминания устазу
  не сделано.
