# Подсчёт пропусков — привязка к группе, не к пользователю (15.07.2026)

## Суть

Пропуски (`get_days_since_last_report`, `get_skip_count_month`,
`get_miss_count_last_30_days` в `core/db.py`) считаются от даты, когда студент стал
активным **именно в этой группе** — `user_groups.joined_date` — а не от
`users.added_date` (дата первой регистрации в боте вообще, глобальная для человека).

```python
def get_skip_count_month(uid, group_id=None):
    ...
    added = user["added_date"]
    if group_id:
        joined = _group_joined_date(uid, group_id)  # user_groups.joined_date
        if joined and joined > added:
            added = joined
    start = max(month_start, added)
```

`add_student(name, group_id, phone)` обновляет `joined_date=today` при КАЖДОЙ
(пере)активации — первая регистрация, ручное восстановление, автовозврат через
дыру авторегистрации (см. [[prep_group]]), перевод из другой группы. Это относится и
к `core/prep.py` — там `count_report_days_since` читает `joined_date` напрямую из SQL,
без обёрточных функций, но тот же `add_student()` держит его свежим.

## Почему это важно

Раньше глобальная `added_date` означала: если человек хотя бы раз пропускал дни в
начале месяца, а потом стал сдавать регулярно, старые пропуски никуда не девались —
счётчик месяца продолжал расти относительно 1-го числа. Комбинация с дырой
авторегистрации (см. [[prep_group]]) создавала бесконечный цикл: кик за пропуски →
студент возвращается в чат → дыра тихо реактивирует его как студента → на утро
scheduler видит тот же старый счётчик ≥ порога → кик снова.

Реальный кейс: Камаль (uid 81, N-1/pro) кикался ботом 4 дня подряд (12–15.07.2026),
хотя фактически сдавал отчёты почти каждый день — просто в начале месяца было 9-10
пропущенных дней, которые никогда не сбрасывались. В тот же день (15.07) та же
причина ударила по Бахадыру (G 2b) — подтверждено, что баг системный.

## Решение пользователя (15.07.2026)

«Если человек повторно зашёл — обнулить, пусть сдаёт заново по правилам группы».
Применено универсально: при входе в ЛЮБУЮ группу, где сдают задания (pro/relaxed),
пропуски считаются заново от даты входа. Разные пороги pro/relaxed (`PRO_INACTIVE_DAYS`,
`RELAXED_INACTIVE_DAYS` в `core/transfers.py`) не затронуты — меняется только точка
отсчёта, не сами лимиты.

Коммит `210bd0a`. Затронутые вызовы: `core/transfers.py` (проверка переводов),
`core/handlers.py` (`/mystats`), `core/scheduler.py` (`personal_reminders`,
`skip_warnings`).
