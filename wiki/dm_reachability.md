# Личные сообщения студентам — ограничение Telegram и как его обходим

## Суть ограничения

Telegram Bot API не даёт боту написать пользователю первым, пока тот сам не
написал боту в личку хотя бы раз (`/start` или любое сообщение). Это
платформенное ограничение против спама, не поведение кода — обойти нельзя.
Ошибка при попытке: `403 Forbidden: bot can't initiate conversation with a
user`.

Из-за этого молча проваливаются все существующие личные рассылки, если
студент никогда не открывал ЛС с ботом: `individual_reminders` (15:00),
`skip_warnings` (20:30), уведомления о переводе в Тадаббур и т.д. — за 7 дней
логи показали 664 таких отказа.

Второе, не связанное ограничение: у части студентов в БД (`users.phone`,
это Telegram user_id, не номер телефона) записан **невалидный id вообще** —
`getChat` возвращает `chat not found`. Аудит 13.07 по 115 активным студентам:
**90 (78%)** id валиден, **25 (22%)** — битый (нужно будет попросить их
прислать `/id` заново и поправить запись).

## Обходной путь — только через групповой чат

Бот **не может** физически добавить студента в группу по user_id (тоже
платформенное ограничение Bot API — только через инвайт-ссылку, см.
`registration_flow.md`) и не может написать в ЛС первым. Единственный канал
достучаться до неподключившегося студента — сообщение в группе, где бот уже
состоит, со ссылкой-приглашением на ЛС: `https://t.me/<bot_username>?start=go`
— по клику Telegram сразу открывает диалог с кнопкой Start, один тап
достаточно и навсегда (Telegram запоминает разрешение, если только студент
потом не заблокирует бота).

## Реализация — `users.dm_ok` + `dm_connect_reminder`

- Новая колонка `users.dm_ok` (миграция в `core/db.py`, дефолт 0).
- `mark_dm_ok_by_phone(phone)` вызывается в начале `process_message`
  (`core/handlers.py`) при **любом** сообщении студента боту в личку — не
  только `/start`, любое сообщение доказывает, что канал открыт.
- `dm_connect_reminder()` (ежедневно 08:00, `core/scheduler.py`) — для каждой
  **не-Тадаббур** группы публикует список ещё не подключившихся студентов
  (`dm_ok=0`) со ссылкой. Группа, где все уже подключены — ничего не
  публикует. Тадаббур сознательно исключён (14.07, устаз заметил лишнюю
  рассылку там — не тот тон/аудитория для операционной нагрузки).
- Юзернейм бота берётся через `getMe` каждый раз, не хардкодится — на случай
  разных username у мужского/женского профиля.

## Публичное упоминание имён в группе → замена на личку (21.07.2026)

`tadabbur_invite_reminder()` (`core/scheduler.py`) раньше постил в группу
одно сообщение со списком имён всех, кто ещё не в Тадаббуре — по просьбе
пользователя это публичное «называние» отстающих заменено на личные
напоминания. Логика: для каждого отстающего — реальная попытка
`send_message(phone, ...)`, проверяется `resp.get("ok")` (не флаг `dm_ok`,
а живой результат отправки — тот же принцип, что ниже в разделе
диагностики). Только те, кому личное не ушло (нет `phone` или Telegram
вернул ошибку — не жали `/start`), попадают в список для группового
сообщения. Если ушли ВСЕМ лично — группе не постится ничего.

## Известный пробел (не закрыт)

`dm_ok` не сбрасывается, если студент **после** первого сообщения боту его
заблокирует — флаг остаётся `True` навсегда, `dm_connect_reminder` перестанет
упоминать студента в группе, хотя ЛС реально снова недоступны. Нужно ловить
ошибку `Forbidden: bot was blocked by the user` при отправке и сбрасывать
`dm_ok=0` в ответ. Не сделано — устаз в курсе, чинить по мере необходимости.

## Диагностика: как проверить достижимость студента вручную

`sendChatAction` — **ненадёжный** прокси для проверки (даёт другие ошибки —
`PEER_ID_INVALID` — чем реальный `sendMessage`, не используй для аудита).
Правильный способ проверить, валиден ли id вообще (без спама реальным
сообщением): `tg_call("getChat", {"chat_id": phone})` — `ok: True` значит id
резолвится (бот видел студента хотя бы в общем чате), `chat not found` —
запись битая. Проверить именно возможность писать в ЛС можно только реальной
попыткой `sendMessage` (или проверкой `users.dm_ok` в БД).
