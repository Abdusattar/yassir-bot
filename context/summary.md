- **Убрали ненадёжный слой из отчёта теневой проверки.** `letter_hint()` в

  После правки слова без конкретного диагноза не показываются вовсе
  (раньше — "стоит перепроверить", тоже убрано по совету эдвайзери).

  справляется с обычной речью). Сравнение с находками пайплайна:

  подтверждает предыдущий негативный контроль); **ихфа — не работает**
  (85% vs 100%, тот же сломанный флаг гунны внутри) — но её и не было в
  отчёте студенту, менять нечего.

- **ИИ-таджвид, продолжение той же сессии — установленный факт: универсальная
  аудио-LLM не может быть судьёй тонкого произношения.** Три независимых
  доказательства сошлись: мой негативный контроль выше; механистическое
  объяснение внешнего эксперта (аудио-LLM сжимают звук в семантические
  токены ~25-50 Гц для понимания смысла, не для акустической
  микроинспекции); индустриальное поле бенчмарков **Mispronunciation
  Detection and Diagnosis (MDD)**, где универсальные аудио-LLM показывают
  <45-55% F1 на уровне фонем. Дополнительно поймано на трёх моделях подряд
  (`gemini-3.1-pro-preview`, `~gemini-pro-latest`, `xiaomi/mimo-v2.5`):
  модель прямым текстом пишет «аудио недоступно» и всё равно уверенно
  выдумывает детали — прямое доказательство, что «уверенно + детально» не
  значит «реально слышала».
  **Найдена рабочая альтернатива**: гибридный пайплайн (не сквозная LLM) —
  forced alignment специализированной моделью + детерминированный DSP-анализ
  (длительность/спектр) + text-LLM только для финального фидбека. Ключевая
  модель **проверена и реальна**: `TBOGamer22/wav2vec2-quran-phonetics`
  (HuggingFace, 349k скачиваний) — запущена локально бесплатно на Бакыте и
  на эталонах Хусари/Афаси, реально распознаёт коранический текст (не
  выдумывает), но сырой вывод без forced alignment бесполезен как есть.
  Технический план из 7 шагов записан (исправлен ревью — добавлен бесплатный
  Гейт №0: тест ложных срабатываний на уже полученных decode'ах
  Хусари/Афаси, должен идти первым при возобновлении темы). **Решение
