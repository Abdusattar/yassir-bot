## Часть 2: недельный операционный отчёт — устаз не может отметить урок сам
- Жалоба устаза на присланный `weekly_ops_report`: «Yassir подготовительная» попала
  в «не провели урок», хотя урок он провёл лично — просто ни один студент не
  отметился «у».
- Причина: `handlers.py:1038` (`u`/`у`) искал студента через `find_by_phone`,
  которая матчит только `role='student'`. У устаза в группе `role='admin'` —
  поэтому его собственное «у» тихо игнорировалось, ни ответа, ни записи в
  `score_events`.
- Заодно вскрылась вторая неточность в том же отчёте: сравнение «к прошлой
  неделе» по проверке таджвида (`voice_pct`) считалось нечестно — колонка
  `voice_submissions.reviewed_at`/трекинг появились только 04.07, а «прошлая
  неделя» для сравнения — 29.06–05.07, то есть только 2 из 7 дней реально
  отслеживались.
- Фикс (коммит `40a86bf`, задеплоен): 1) устаз, не найденный как студент, но
  являющийся `is_group_admin` этой группы — засчитывается как «урок проведён»
  (`score_events`, category='attendance', subcategory='ustaz', 0 баллов, отдельное
  сообщение `T("lesson_marked")` — без «+5 баллов», это не бонус); 2) в
  `weekly_ops_report` сравнение с прошлой неделей по каждой метрике (уроки/задания/
  таджвид) теперь скрывается, если трекинг этой метрики не покрывал предыдущую
  неделю целиком (`_metric_tracked_since`) — без хардкода дат, самовосстанавливается.
- Проверено сценарным тестом (временная БД): устаз пишет «у» → приходит
  «✅ Урок отмечен как проведённый», `_week_ops_stats` засчитывает группу, баллы
  не начисляются.
- Ретроактивная правка данных: устаз подтвердил, что провёл урок в подготовительной
  12.07 (вчера). Вручную добавлена запись `score_events` за 12.07 (`add_bonus`,
  устаз id=8 «Нурсултан», group_id=14) — разовый backfill на проде, не через код.
- Уже отправленный `weekly_ops_report` за 06.07–12.07 переотправлен вручную с
  пометкой «(уточнено)»: подготовительная убрана из «не провели урок» (7→8 групп),
  сравнение по таджвиду с прошлой неделей убрано как нечестное. Устаз подтвердил
  текст перед отправкой.
- Попутно: узнали, что `.env` подхватывается только через `systemd
  EnvironmentFile` — при ручном запуске скриптов по SSH токен пустой («Not Found»
  от Telegram API), нужно `set -a; source .env; set +a` вручную.
- Попутно объяснили устазу «(?)» в ежедневном отчёте по голосовым — `sent_at`
  добавили только 12.07 без бэкфилла, у сдач за 11.07 и раньше время просто не
  сохранялось. Не баг, само уйдёт.

## Часть 3: физический кик при переводе + Ринат/Бахтияр в G-6 + дыра «bot can't initiate conversation»
- Живой кейс: Айдин (N-2a, pro) переведён в Тадаббур 12.07 за 10 дней без отчёта
  (`student_transfers`), но вчера же написал в старом чате N-2a → дыра
  авторегистрации вернула его туда активным студентом одновременно с Тадаббуром.
  Причина глубже вчерашней: `deactivate_student()` — чисто `UPDATE user_groups
  SET active=0`, физически из Telegram-чата никого не кикает, студент как был
  участником чата, так и остаётся — сообщение «переведён» вводит в заблуждение.
- Фикс (коммит `335564e`, задеплоен): `_transfer_to_tadabbur()` в
  `core/transfers.py` теперь сразу после `deactivate_student()` делает мягкий
  кик (`ban_member`+`unban_member`, паттерн как в `kick_unregistered()`) — и для
  причины «неактивность», и для «пропуски уроков». `_suggest_upgrade()` не тронут.
  Не чинит задним числом уже сломанные случаи (Айдин/Медер/Камал уже физически в
  старых чатах) — закрывает дыру только для будущих переводов.
- Уточнили дизайн: по `transfers.py` header — `tadabbur → relaxed` (по желанию
  студента) — единственный документированный путь возврата, независимо от того,
  ушёл человек из pro или relaxed. Автоматического кода для возврата нет, решает
  устаз вручную.
- Подтвердили: устаз добавлял их 5 (pro) / 15 (relaxed) дней до порога (`skip_warnings`,
  20:30) — предупреждение с правилом и «сколько дней осталось» шлётся в ЛС
  ежедневно начиная с этого порога; фраза «вернёшься когда захочешь» — из другого,
  более позднего сообщения (`transfer_to_tadabbur`, уже после самого перевода).
- Попытка выполнить план «Ринат/Бахтияр → G-6» (id=13, invite_link уже есть):
  `add_prep_graduate()` для обоих выполнен (заработает автоматически при входе по
  ссылке через уже задеплоенный `announce_prep_graduate_arrival`), но отправка
  ссылки личным сообщением не прошла ни одному:
  - Ринат (id=105) — id валиден (`getChat` ok), но Telegram запретил
    («Forbidden: bot can't initiate conversation») — никогда не писал боту в ЛС.
  - Бахтияр (id=78) — id вообще не резолвится («chat not found») — битая запись.
  Устазу нужно прислать ссылку `https://t.me/+HXpXQwB1tU9kYTAy` лично.
- Это вскрыло системную проблему: аудит (`getChat`) по всем 115 активным
  студентам с телефоном — у **90 (78%)** id валиден, у **25 (22%)** — битый,
  никогда не резолвился в Telegram вообще (нужно будет позже собрать список и
  переспросить их id через /id). Из 90 валидных неизвестное число не ЛС-доступны
  просто потому что не писали боту в личку — за 7 дней логи показали 664 случая
  «can't initiate conversation» в реальных рассылках (напоминания/предупреждения).
- Решение (согласовано с устазом): раз бот не может написать первым тем, кто не
  подключился, — единственный канал достучаться до них это групповой чат. Добавлена
  ежедневная задача `dm_connect_reminder()` (08:00, `core/scheduler.py`):
  - Новая колонка `users.dm_ok` (миграция в `db.py`), выставляется в `True`
    (`mark_dm_ok_by_phone`) при любом первом сообщении студента боту в личку
    (`core/handlers.py`, в начале `process_message`)
  - `dm_connect_reminder()` для каждой группы публикует список имён тех, кто ещё
    не подключился, со ссылкой `https://t.me/<username>?start=go` (бот-юзернейм
    берётся через `getMe`, не хардкод — на случай разных username у муж/жен бота);
    группа, где все уже подключены — ничего не публикует
  - Проверено сценарным тестом (изолированная БД, замоканы send_message/tg_call):
    после группового сообщения `dm_ok` не меняется, после ЛС — становится True,
    напоминание упоминает только неподключившихся
  - Коммит `03321f4`, задеплоено.
