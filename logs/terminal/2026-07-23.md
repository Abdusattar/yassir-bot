## Часть A — баги/улучшения бота (задеплоено на оба бота)

- Разобрана путаница у Арзу (зачёт + перевод в Тадаббур почти одновременно) →
  честная формулировка перевода с датами и долей сданных дней вместо мутного
  «за N дней без отчётов».
- Найдены протухшие `chat_id` в 6 группах (миграция в supergroup) — обновлены
  на сервере (Отбор., G-4, G-7, G-9, G-12, Группа для детей).
- Реализовано правило: возврат в постоянную группу только через успешное
  прохождение подготовительной — и для кикнутых за пропуски, и для СОВСЕМ
  новых студентов. Маркер `pending_prep_return` (не факт членства в Тадаббуре
  — там все студенты). Единый критерий `prep_days_done()`.
- Автоссылка при выпуске из подготовительной убрана — вместо неё уведомление
  ОДНОМУ устазу (Умар устаз / Зейнеб на женском боте), чтобы не разобрали
  одного студента в две группы одновременно. Устаз сам определяет группу,
  бот сам кикает из prep, когда видит студента активным в новой группе.
  Добавлен публичный пост в саму подготовительную при успехе (мотивация),
  и возможность студенту самому напомнить устазу, просто написав боту.
- Уведомление об опустошении очереди `curriculum_parts` (нахв/таджвид) за
  неделю до конца — в личку SUPER_ADMIN_IDS[0].
- Скопированы недостающие уроки нахва/таджвида в `quran_female.db`.

## Часть B — R&D: собственная проверка таджвида

**Гейт №0, вторая попытка (после провала TBOGamer22, 89% ложных срабатываний,
см. вчерашний лог)**: отправлен запрос трём разным ИИ на поиск моделей получше
(с полным контекстом провала). Все три ответа перепроверены по первоисточникам
(arXiv, HF, GitHub) — почти всё реальное, кроме одной натяжки третьей модели
(Quran-MD ошибочно названа "phoneme-level", по факту только word-level).

Главный вывод, снимающий разногласие между источниками: у всех трёх один и
тот же шов — уровень транскрипции/слов решён (готовые Whisper fine-tunes),
уровень артикуляционного суждения (то, что нам реально нужно) — открыт.

**Прогон obadx/quran-muaalem** (по просьбе — сразу, не вторым приоритетом):
скачаны реальные веса (`obadx/muaalem-model-v3_2`), референсные параметры
чтецов (`MoshafAttributes`) взяты не наугад, а из метаданных датасета
obadx (Хусари = moshaf 0.0, Афаси = 19.0). Результат на control-аудио
(Хусари+Афаси, аяты 106-112):
- Phoneme-level: **95.8%** точных совпадений (было 2/19 у TBOGamer22).
- Sifat-level (среди совпавших фонем): **0.2%** расхождений.
- Sanity-check (аудио с чужим эталоном) подтвердил, что декодирование
  реально идёт из звука, не подглядывает в эталон.
- **Важная оговорка** (поймана вторым заходом к ревьюеру): это тест только
  specificity (не путается на верном чтении). Sensitivity (ловит ли
  настоящую ошибку) не проверена вообще — судья, слепой к ошибкам, тоже
  дал бы ~0% здесь. Не путать "категорически лучше TBOGamer22" с "пайплайн
  работает как детектор".

Всё записано в `wiki/ai_tajweed_audio.md`.

**QDAT** — датасет с реальными correct/incorrect метками (Мосульский
университет, 2021, arXiv:2305.06429) — единственный найденный источник
с настоящими ошибками, нужен для проверки sensitivity. Найден первоисточник
на Kaggle (`annealdahi/quran-recitation`) — 1500 записей WhatsApp-начиток
(не профессиональные чтецы, есть дети), 11кГц, 3 правила (Мадд мунфасиль,
Гунна, Ихфа), 40% верных. Скачивание анонимно без Kaggle-аккаунта началось
(~1.37 ГБ, медленно, ~2.5-3ч) — оставлено в фоне, разбор и прогон в
следующей сессии.

## Часть B, продолжение (вторая половина дня) — от QDAT-пилота к Stage 0

**Убран ASR-буквенный слой из отчёта теневой проверки.** По прямому запросу
пользователя ("эдвайзери пусть проверит, все ли годится для того чтобы
студента поправлять") consult с advisor выявил: `letter_hint()` в
`check_voices.py` строил "буква X вместо Y" не на Muaalem (провалидированная
модель), а на сравнении с транскрипцией обычного tarteel-Whisper — того же
ASR, что сам путает гортанные буквы у начинающих. Негативный контроль на
Хусари (аяты 106-109, заведомо верное чтение) подтвердил эмпирически:
пайплайн выдумал "поправку" `ٱ → ا` (орфографический вариант алифа, не
ошибка произношения) на 109-м аяте. `letter_hint`/`build_word_hints`/
`LETTER_EXPLANATIONS` удалены из кода. Осталось только `rule_hint` (мадд-
длина + калькаля, из Muaalem) — провалидированный сигнал.

**Живое сравнение с устазом.** Пользователь прислал голосовое Умар устаза —
реальную коррекцию на свою же запись (аят 2:77). `tarteel-ai/whisper-base-
ar-quran` не справился (не Коран, обычная речь) → расшифровано обычным
`openai/whisper-small` (уже был скачан на компьютере пользователя):
"...يَعْلَمُ مَا يُسِرُّونَ — мим мы потянули... И также يُسِرُّونَ —
остановку сделали на нун. А здесь должен быть يُسِرُّونَ, либо остановка на
сукуне, либо соединяем." Пайплайн на этой же записи отметил те же 2 слова
(يَعْلَمُ, يُسِرُّونَ) из своих 4 — но без диагноза первопричины (модель
среагировала на букву ع в других словах, устаз говорил про мим-растяжку и
васль/вакф на يُسِرُّونَ). Advisor поправил первую формулировку
("валидация") на честную — "совпадение по месту, n=1, причины не сравнивались".

**Убрана и вторая правка формата отчёта.** По совету advisor "стоит
перепроверить" (для слов без rule_hint) тоже убрано — теперь показываются
только слова с конкретным диагнозом мадд/калькаля, остальные молча
пропускаются (честнее, чем placeholder ни о чём не говорящий).

**QDAT — гунна закрыта окончательно, мадд/ихфа получили честные цифры.**
Пользователь передумал насчёт "не гнаться за моделью" ("если что-то отложил
а оно нужно — беру слова обратно") и попросил докрутить методологически
правильно. Advisor посоветовал сначала прочитать CSV-метки (минуты, без
модели) и посчитать "чистые кроме X" подмножества, прежде чем гнать дорогой
прогон:
- Чисто кроме гунны (мадд=1,ихфа=1,гунна=0): **0 из 1509**. Структурная
  корреляция меток на ВСЁМ датасете, не артефакт пилота n=19 — порог для
  гунны учить не на чем, точка. Гунна остаётся выключена насовсем.
- Чисто кроме мадда: 151. Чисто кроме ихфы: 199. Плюс контроль (все три
  верны) — подвыборка 300. Прогон 620 файлов (`tajweed_ai/scripts/
  qdat_clean_subset_run.py`, ~2.8с/файл на CPU, фоном):
  - **Мадд**: sensitivity 41.8% (56/134 верно поймано), specificity 97.3%
    (290/298 не спутано) — порог `pred_madd_len <= 2`. Реальный, но слабый
    сигнал; согласуется с ложными срабатываниями на Хусари выше.
  - **Ихфа**: 85% "есть ихфа" при true_ikhfa=0 vs 100% при true_ikhfa=1 —
    разница 15 п.п., практически не различает. Причина — `ikhfa_detected()`
    в `qdat_run.py` сам опирается на `ghonna=='maghnoon'`, тот же сломанный
    флаг. Ихфы и так нет в отчёте студенту (`rule_hint` проверяет только
    мадд+калькалю) — менять нечего, но добавлять её нельзя без другого
    метода детекции.

**Вопрос пользователя "как приблизиться к устазу" — advisor разграничил
дёшево/дорого.** Гунна (порог по `.prob`, а не по бинарной метке — такое
поле есть у `SingleUnit`) технически дешевле retrain, но реально невозможна
без нового датасета (см. 0 выше). Васль/вакф — не измеряется Muaalem
вообще: эталонная фонетизация аята всегда строит "продолжаем" (нет
альтернативного "с сукуном" эталона для сравнения) — нужна отдельная фича
(два эталона на слово), и концептуально шатко (остановка часто законный
выбор, не ошибка). Оставлено нетронутым, решение — пользователя.

**Пивот на Stage 0 (реальные корни данных, не чужие датасеты).**
Пользователь: "давай так, если я что-то отложил а оно нужно то я беру свои
слова обратно" + сообщил квалификацию Умар устаза (25 лет с Кораном, хафиз,
образование по Корану в Университете Медины — записано в память Claude
Code, `project_umar_ustaz_credentials`) — обосновывает, почему именно его
коррекции стоит копить как ground truth, а не силами QDAT/Хусари (чужие
чтецы/студенты).

Advisor до имплементации указал: разведка кода важнее гипотез — оказалось,
`mark_voice_reviewed()` хранит только `reviewed_at` (булево "проверено"),
содержимое реплая устаза НИКОГДА не сохранялось (`verify_log` — про
текстовый нахв/муфрадат, не туда). То есть исторического архива нет,
копить можно только с нуля, вперёд.

Пользователь уточнил важный scope: не любой админ группы (как сейчас в
`is_group_admin`), а **именно Умар устаз**. Найден его идентификатор —
`CURRICULUM_REVIEWER_ID` (config.py, уже используется как "второй
SUPER_ADMIN_ID, устаз Умар").

**Реализовано и задеплоено (коммит `1719165`):**
- `core/db.py`: `voice_submissions` +3 колонки (`review_type`,
  `review_file_id`, `review_text`); новая `save_umar_review()` — пишет
  ТОЛЬКО когда `phone == CURRICULUM_REVIEWER_ID`, без автотранскрипции.
- `core/handlers.py`: рядом с существующим `mark_voice_reviewed()` (реплай

  пары (запись студента + коррекция Умара, если голосовая) в

  авто-удаляются — "поучились" не машинный триггер; чистка только через
  сознательный `--purge-reviewed` по вручную отмеченным id.
