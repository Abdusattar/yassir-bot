## Вечер — рефакторинг архитектуры данных

### Новая схема: users + user_groups
**Проблема**: `students` хранил личность и членство вместе (group_id внутри). Повторная регистрация при вступлении в новую группу.

**Решение**:
- `users(id, name, phone, active, added_date)` — глобальный реестр личности
- `user_groups(user_id, group_id, role, active, joined_date)` — членство (role: student/admin)
- Студент в ОДНОЙ учебной группе (pro/relaxed), может быть в Тадаббур, может быть admin в других
- `group_admins` таблица вытеснена в `user_groups` с role='admin'

**Миграция**: автоматически при старте бота через `_run_migrations` → `_migrate_to_users_table`
- students → users + user_groups
- group_admins → user_groups (role='admin')
- reports.sid перемаплен на users.id (с merge при конфликтах через MAX)

**Ключевые изменения API**:
- `deactivate_student(uid, group_id)` — деактивирует членство в конкретной группе
- `find_user_by_phone(phone)` — глобальный поиск без привязки к группе
- При вступлении в новую группу: если пользователь уже в users → авторегистрация молча
- Весь SQL JOIN через users + user_groups (не students)
